# The Invariants

**Evidence tier:** [I]
*Organism document. Interpretive operational content. Bounded by current system state.*

**Rosetta boundary:** [I] This compressed invariant map orients K/L/O routing. It does not [B] authorize new invariants or collapse namespaces without source-lane receipts.


## The three layered namespaces

Post-rename, there are three distinct invariant families:

- `K0–K9` = **DAC-constitutional** layer
- `L0–L9` = **kernel / hashgraph** layer
- `O0–O4` = **OFN ecosystem protocol** layer

Do not collapse them into one namespace.

## The K layer — organism law

The most important K-laws to carry:

- `K0` — No Receipt, No Reality
- `K2` — Mortal Substrate
- `K4` — Grace Exit
- `K6` — Zero Extraction

These are not optional values. They are the core runtime constitution.

## The L layer — kernel law

Useful anchor points:

- `L0` — Autopoiesis
- `L4` — the L-function / collateral physics layer
- `L6` — checkpoint commitment
- `L8` — no native delegation

The L-layer is where the kernel keeps the organism alive and ordered.

## The O layer — OFN law

Useful anchor points:

- `O0` — Right to Resell
- `O1` — Royalty Persistence
- `O2` — Grace Exit at commerce layer
- `O4` — `eta = 0` at protocol layer

OFN is intentionally smaller than the full K-set. It is a compatibility
surface for issuers, not the full DAC constitution.

## The compile rule

If a proposed invariant cannot be:

1. traced to a Foundation source
2. stated non-redundantly
3. checked against existing layers

then it should not be added.

## What matters practically

For most real decisions:

- if it affects governance or exit, think `K`
- if it affects kernel/settlement mechanics, think `L`
- if it affects issuer compatibility or asset behavior, think `O`
