# Extension Gate Checklist

**Evidence tier:** [I]  
*Organism document. Interpretive operational content. Bounded by current system state.*


> **No extension may enter the organism unless it tightens at least two flywheels without weakening K2 or η = 0.**

Date: 2026-04-16

## 0. Purpose

The final gate is the most dangerous.

Once the core organism breathes, the temptation to add surfaces becomes constant: new products, new chains, new narratives, new markets. This checklist exists to make every extension earn its place structurally — not narratively.

---

## 1. The One-Sentence Law

**An extension is lawful only if the organism would be weaker without it.**

Not "bigger." Not "more impressive."  
**Weaker without it.**

---

## 2. The Six Flywheels

Any proposed extension must explicitly name which flywheels it strengthens:

1. **Proof flywheel** — action → receipt → accounting → statement → proof → trust
2. **Graph flywheel** — relations → graph capital → cheaper money → more activity → more relations
3. **Capital flywheel** — ZAI base → SKY flow → activity → receipts → better pricing → disciplined issuance
4. **Infrastructure flywheel** — edge nodes → uptime → auditability → graph capital → capability → revenue → infrastructure
5. **Contagion flywheel** — bad behavior → penalty → quarantine → cleaner graph → better pricing → resilience
6. **Cognition flywheel** — receipts/graph → nervous system → anomaly/opportunity detection → better action → better receipts

An extension that does not map clearly to at least **two** of these is scope creep.

---

## 3. The Extension Gate Checklist

### Section A: Structural Fit

| # | Question | Pass Criteria |
|---|----------|---------------|
| A1 | Which two flywheels does this extension directly strengthen? | Must name at least two, with causal explanation |
| A2 | Does the extension require any new Layer 1 feature? | If yes, explain why existing event envelopes cannot carry it |
| A3 | Does the extension introduce a new DAC type or modify an existing one? | If yes, specify how audit membrane and OFN coverage apply |
| A4 | Does the extension require a new SoResFi tool? | If yes, specify unlock criteria (`c`, `x`, `t`, receipt coverage) |
| A5 | Does the extension create a new revenue stream? | If yes, map every fee to `32_FEE_DOCTRINE.md` |

### Section B: K2 and Sovereignty

| # | Question | Pass Criteria |
|---|----------|---------------|
| B1 | Does any execution path in the extension bypass K2? | Must be NO. Any exception must be a protective halt, not an active execution |
| B2 | Does the extension collect identity or biometric data beyond NEXUS D1–D6? | Must be NO |
| B3 | Does the extension create a new signing key or custodial arrangement? | Must be NO unless it strengthens K2 (e.g. multi-sig, backup) |
| B4 | Does the extension make Grace Exit harder or more expensive? | Must be NO |

### Section C: η = 0 Discipline

| # | Question | Pass Criteria |
|---|----------|---------------|
| C1 | Does the extension introduce rent-like pricing (pay for access, not value)? | Must be NO |
| C2 | Does the extension capture value from graph capital or connectivity scoring? | Must be NO — `c(DAC)` is a public metric, not a SKU |
| C3 | Does the extension create a platform lock-in effect? | Must be NO — data must remain portable, exit must remain frictionless |
| C4 | Does the extension enrich Skyzai from user failure or conflict? | Must be NO — no penalty harvesting, no dispute profit |

### Section D: Proof and Receipting

| # | Question | Pass Criteria |
|---|----------|---------------|
| D1 | Can the extension's core action produce an OFN receipt? | Must be YES |
| D2 | Does the extension have a live statement path? | Must be YES, or must explain why it is exempt |
| D3 | Does the extension update the NEXUS graph or DAC connectivity? | If yes, specify how. If no, explain why it is graph-invisible |
| D4 | Is there a 30-day operational proof plan before public launch? | Must be YES |

### Section E: Scale and Scope Discipline

| # | Question | Pass Criteria |
|---|----------|---------------|
| E1 | Does the extension duplicate an existing organ's constitutional role? | Must be NO — no organ creep |
| E2 | Does the extension require a new token or inflation mechanism? | Must be NO — LP-100 and SKY elasticity are sufficient |
| E3 | Does the extension expand the Constitutional Grand Sequence out of order? | Must be NO — must respect proof-before-intelligence-before-extensions |
| E4 | If the extension fails, can it be cleanly removed without organism damage? | Must be YES — extensions must be modular, not load-bearing |
