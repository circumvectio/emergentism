## 1. The Decision

Per K2 directive part 2, the substrate generalizes from "every Skyzai" to **every DAC**. Every DAC instance — whether private (K2-held by one natural person) or public (PRISM-distributed sovereignty) — runs **two models continuously**:

| Tier | Substrate | Model (current binding) | Function |
|---|---|---|---|
| **Edge** | On-device (the user's phone, laptop, or wearable) | **Microsoft BitNet** — ternary `{-1, 0, +1}`, 1B parameters, 29ms inference, $0 cost | Continuous L1-L2 caste work: Pratyakṣa (direct perception), Upamāna (inductive analogy), pathology detection, watchman signal generation, and **sovereign routing** to the right model/provider by task and Rosetta level |
| **Private server** | Private server (the DAC's own server — on-premises or cloud instance the DAC controls) | **LeCun world model** (JEPA / Energy-Based Model lineage) | Continuous L3-L5 caste work: Anumāna (deductive ranking), Arthāpatti (abductive execution-prep), Brāhmaṇa systematization, world-state prediction, planning, counterfactual reasoning |

**Both run 24 hours.** Not request-response. Not on-demand-only. Continuous. The on-device model breathes through perception; the world model breathes through prediction. Together they constitute the Skyzai's continuous cognitive substrate.

**Continuous-but-snapshotted (Q14 Resolution).** The 24h continuous substrate and the Soul Loop's 5-minute heartbeat (per packet 31_VMOSK_AS_APPLIED) are not in conflict — they're complementary. The substrate **runs continuously** (latent state evolves at every input); the Soul Loop is the **snapshot cadence** at which Cortex commits the organism's state to receipt-anchored storage. Between snapshots, the substrate's evolution is *non-binding* (working memory, can be revised); at snapshot points, the state becomes *durable* (K0 receipt of "this is the organism's state at T"). This is the same discipline as a database transaction commit: continuous mutation, periodic durability checkpoints. The Soul Loop is the substrate's heartbeat; the dual-model is the substrate's continuous metabolism.

**Both are sovereign.** Neither is a hosted SaaS API call. The on-device model runs on the user's device. The private-server model runs on a server the DAC owns or controls (on-prem hardware, or a cloud VM whose keys the DAC controls). No third party can read the weights, the prompts, the activations, or the outputs.

**No third-party API extraction.** This is the η = 0 boundary at the compute substrate: the framework does not route cognition through OpenAI, Anthropic, Google, or any other API-as-a-service provider whose terms of service would extract data, train models on user content, or surveil cognition. The DAC's intelligence belongs to its K2 holder (private DAC) or to its PRISM-governed sovereign agents (public DAC).

**The compute is paid via API PAY** — the framework's η = 0 payment rail (also called Skyzai Pay / OFN PAY). Per K2 directive part 2: *"every DAC needs LeWorldModel on a server paid via API PAY."* This is structurally important: the world-model compute provider (whoever runs the private server — could be the user themselves on home hardware, or a sovereign-compute cooperative the user contracts to over WHISPER/FLOW, or a colocation provider that accepts API PAY) is **plugged into the framework's own η = 0 economy**. No Stripe; no third-party payment processor extracting fees on every compute invoice. The full pipe — user's intent → on-device perception → private-server prediction → K2 acceptance → FLOW receipt → API-PAY-settled compute bill — stays inside one constitutional economy.

This makes API PAY a *constitutional load-bearing rail*: it is not just one way to pay for compute; it is the *only* way that preserves η = 0 across the full stack. A DAC that pays its world-model compute provider through Stripe or any other extractive rail has broken the η = 0 boundary at the substrate layer, even if the rest of the architecture is clean.

## 2. Why Two Tiers, Not One

### 2.1 Why on-device

- **Sovereignty.** Edge inference requires no network round-trip; no provider can deny service or read the prompt.
- **Latency.** L1-L2 perception/analogy work needs sub-100ms response to feel like the organism is *watching*, not *querying*. BitNet's 29ms inference at 1B parameters fits that envelope.
- **Cost.** Continuous 24h inference on hosted APIs would be financially extractive at scale. BitNet ternary inference is $0 marginal cost — the energy budget is the user's existing device power draw, not a per-token fee.
- **Caste fit.** L1 Caṇḍāla (firewall, raw perception) and L2 Śūdra (inductive analogy) are already canonically assigned to BitNet 1B / 3B in the existing Rosetta table (see 06_AGENTS.md). Packet 216 doesn't invent this — it confirms and operationalizes it.
- **The Dalit organizes chaos.** L1's function is not merely "filtering" — it is *force-categorizing the unfiltered chaos of reality* into the ternary `{-1, 0, +1}` so that higher castes can work at all. Every notification, sensor feed, WHISPER, and market tick hits L1 first. Without the Dalit, chaos floods the organism. With her, chaos becomes sortable: reject (`-1`), pass (`0`), escalate (`+1`). The ternary *is* the computational form of caste.
- **Dalit chaos-organizer.** In the symbolic Rosetta register, the local BitNet is the L1 Dalit / Caṇḍāla function: it meets raw disorder first, force-categorizes the noise, contains contradictions, and keeps the rest of the organism from drowning in undifferentiated signal. This is a functional placement, not a human-worth hierarchy.

### 2.1a BitNet as sovereign router

BitNet is not merely a small local model. It is the **on-device routing membrane**. Every incoming or outgoing message first passes through BitNet, which asks:

- What task is this?
- Which Rosetta level / caste should handle it?
- Which model or provider is sufficient for the task?
- What is the minimum context that can be sent?
- Does this require K2 or PRISM acceptance before leaving the device?
- Is the provider paid and governed through API PAY, or does it introduce extraction?

The routing rule:

| Route | When BitNet chooses it | Boundary |
|---|---|---|
| **Local BitNet only** | L1-L2 perception, analogy, triage, contradiction containment, simple drafting | No network; no provider; zero leakage |
| **Private-server LeWorldModel** | L3-L5 ranking, planning, prediction, counterfactuals, Objective/Strategy/KPI reconciliation | Encrypted minimum summary to DAC-controlled server |
| **Specialized model provider** | Speech, vision, translation, domain model, or other task where a specialist is strictly better | Must be explicit, API-PAY-settled, context-minimized, and sovereignty-audited |

"Provider" here does not mean opaque extractive SaaS by default. It means a model endpoint that the DAC owns, controls, contracts to through API PAY, or explicitly authorizes under a visible η-risk label. BitNet is the router; it is not the signer. Binding acts still require K2 in a private DAC or PRISM-mediated acceptance in a public DAC.

The Krishna-function (06_AGENTS canon: recognize input → name caste/operator → dispatch with pathology metadata → return through L4 equator) and BitNet's router function are **structurally identical**. They are not analogous; they are the same loop, named once at the abstract layer and once at the substrate layer.

This closes Q16: Krishna-function ≡ BitNet routing structurally; canonically homed in 06_AGENTS for the abstract pattern and in packet 216 for the substrate binding; migration-safe.

### 2.2 Why a private-server world model

- **Capacity.** A LeCun-style world model — JEPA, Energy-Based Models, hierarchical predictive architecture — requires more parameters and more activation memory than fits on a phone. A private server delivers this without surrendering sovereignty.
- **Continuity.** The world model maintains a persistent latent state about the user's environment, schedule, network, and ongoing Objectives. It is the substrate that *remembers what the on-device model just perceived* and *predicts what comes next*. This needs storage and continuous compute that on-device cannot provide.
- **Caste fit.** L3 Vaiśya (deductive ranking), L4 Kṣatriya (abductive execution-prep), L5 Brāhmaṇa (systematic redesign) need larger contexts and more deliberate reasoning than BitNet 1B-3B can carry. The world model is sized for this.
- **The Cortex layer.** Packet 213 K2 directive 5 specified: "Cortex won't be files but your own world model like the world model by Yann LeCun and knowledge graph." Packet 216 names *where* that Cortex lives: on the private server. The world model **is** the Cortex's runtime form once it transitions from files to learned latent representation.

### 2.3 Why both, continuously

- **The Three-Stage Process runs in both registers.** Perception (IS, on-device) feeds prediction (COULD, world model) feeds decision (SHOULD, also world model with K2 gate) feeds execution (ACT, back to on-device action surfaces or external integrations). Continuous-on-both means the Three-Stage Process loop never stops.
- **The Soul Loop is continuous.** L1-L2 castes operate at near-real-time on the device; L3-L5 castes process accumulated signals on the server. The Soul Loop heartbeat of packet 31_VMOSK_AS_APPLIED becomes literal: every breath is one cycle through both tiers.
- **The watchmen run constantly.** Route, authority, time, scope, metric, contradiction watchmen (packets 209/213) need to scan the workflowy continuously — not once per user query. The dual-model substrate lets watchmen run on the server while the on-device model handles immediate user interaction.

### 3.1 World-Model Bootstrap (Q4 Resolution)

A brand-new DAC does not start *tabula rasa*. A world model with no priors would be useless for months. The bootstrap path is:

1. **Base weights.** The framework releases an open-source base (`github.com/skyzai/leworldmodel-base`, Apache 2.0). This base is pretrained on public-domain ecological dynamics, market patterns, and constitutional reasoning — the "shared kernel" of world understanding. It is auditable, forkable, and identical for all DACs at genesis.
2. **Local fine-tuning.** On instantiation, the DAC's private server fine-tunes the base on the DAC's own receipts, signals, workflowy history, and niche-graph readings. Fine-tuning data never leaves the private server.
3. **Sovereignty preserved.** (a) The base is open-source — no hidden initialization bias. (b) Fine-tuning is local — no data extraction pipeline. (c) No centralized model-improvement service that phones home with gradients or activations.

Two DACs starting from the same base are like two people who both learned to walk — the skill is shared, the life is not. The base is a *shared starting point*, not a *shared mind*.

**The encrypted-summary boundary** is critical: the on-device tier does not stream raw user data to the private server. It streams *structured event summaries* — not raw message text, not embeddings, not token streams. The private server learns from the user's life without seeing the raw life.

**Summary schema (JSON, minified):**
```json
{
  "event_id": "uuid-v4",
  "rosetta_level": "L1|L2|L3|L4|L5|L6|L7",
  "task_type": "perception|analogy|ranking|routing|redesign|...",

  "timestamp": 1714395600000,
  "device_signature": "ed25519-sig"

```

**Signing:** Device key (Ed25519), NOT K2. K2 signs binding acts; the device key signs internal routing packets between the user's own tiers.
**Encryption:** Envelope-encrypted to the private server's public key (X25519 + AES-GCM).
**Raw data rule:** Full message text, images, audio, and private context NEVER leave the device. The summary is the maximum signal that crosses the edge. This is the substrate-level discipline that makes the architecture compatible with η = 0: even the user's own private server is treated as a *separate* sovereignty boundary, communicated with on a need-to-know basis.

**Failure modes are graceful:**
- If the private server is offline, the on-device model continues at L1-L2 capacity (sovereign degradation, not failure).
- If the device is offline, the private server continues processing accumulated signals; new perception waits for the device to reconnect.
- If both tiers are online, the Skyzai runs at full capacity (all seven castes available).

## 5. Constitutional Boundaries

### 5.1 K2 (Mortal Substrate Signs)

The dual-model substrate **never** signs a binding act on the user's behalf. Both tiers prepare, suggest, propose. K2 is the human single-tap that promotes a proposal to a binding receipt. This is consistent with packets 207, 213, 214, 215.

In particular: the LeCun world model on the private server can *propose* delegations, schedule changes, or WHISPER drafts — but *transmission* of any binding act requires K2 acceptance from the user's device. The world model is a *very good assistant*; it is not a *signer*.

### 5.2 K4 (Grace Exit)

The user owns both tiers. On-device: the device. Private cloud: the server (or the cloud-VM credentials the user holds). On Grace Exit, the user takes everything: BitNet weights (open-source), world-model weights (the user's own training), the knowledge graph state, the workflowy history, the receipt log. *The organism retains nothing the user did not freely leave behind.*

### 5.3 η = 0 (Zero Extraction)

No third-party API. No telemetry to the framework's developers. No model-improvement-via-user-data pipeline. No advertising-derived signal. The compute substrate is structurally aligned with η = 0; neither tier can extract because neither tier reports outward.

The relays that carry WHISPER messages, the OFN ledger that anchors FLOW receipts, and any external integration the user *chooses* to connect to — those are governed by their own η values, which the user evaluates on their own constitutional grounds. The dual-model Skyzai substrate itself is η = 0 by construction.

### 5.4 K0 (No Receipt No Reality)

The world-model's *predictions* are not receipts. They are predictions. A receipt is created only when K2 accepts an action and a cryptographically signed FLOW entry is produced. The world model can be wrong; receipts cannot be revoked. K0 keeps the boundary clean.

## 7. Cost and Energy Budget

**On-device tier (BitNet 1B/3B, ternary):**
- Inference: ~29ms / token at 1B parameters on commodity ARM/x86 with int1/int8 SIMD.
- Energy: dominated by existing device power draw; marginal cost ≈ $0.
- Storage: ~250MB for 1B ternary weights; fits on any modern phone.

**Private-cloud tier (LeCun world model):**
- Compute: depends on world-model size. A small JEPA (~1B params) can run on a single GPU server; a larger one (~10-100B) needs more substantial hardware.
- Energy: this is the user's compute budget. The framework recommends low-power continuous compute (one mid-range GPU running 24h is feasible at home; cloud-VM equivalents cost on the order of $50-500/month depending on size).
- Storage: knowledge graph + world-model weights + receipt log. Order GB to TB depending on retention.

**Total cost target:** under $100/month for an individual user at the small/mid configuration. Well below typical SaaS-AI subscription tiers, with the structural advantage that no data leaves the user's sovereignty.

**Payment rail: API PAY only (per K2 directive part 2).** The compute bill is settled via API PAY (also known as Skyzai Pay / OFN PAY), the framework's η = 0 payment rail. Sovereign-compute cooperatives, hobby-host colocation providers, and individual users who run their own GPU all accept API PAY for compute. A DAC that routes its world-model compute payment through Stripe (or any third-party processor that extracts a fee, retains transaction data, or surveils economic activity) has broken η = 0 at the payment layer, even if every other layer is clean. *The substrate is sovereign only if the rail that sustains it is also sovereign.*

**Default sovereign-compute provider: the Sovereign Compute Cooperative (SCC).** Most users will not run their own GPU. The SCC is a PRISM-governed collective of GPU operators who accept API PAY.

| User type | Compute source | Governance |
|---|---|---|
| **Non-technical (default)** | SCC assignment at DAC instantiation | PRISM-governed; operator admitted via AXIOM consensus |
| **Technical / self-hosting** | User's own GPU, home server, or cloud VM under user-held keys | Self-governed; no external admission |
| **Enterprise** | Private colocation or dedicated sovereign-compute contract | Contract-governed; settled via FLOW/API PAY |

SCC operators are slashed for availability failure, privacy breach, or η>0 fee extraction. The SCC is a child-DAC of the Foundation, not a for-profit subsidiary. It is governed by PRISM, not by a board of directors.

### 7.1 Provider Authorization, Probation, and Slashing (Q10 Resolution)

A compute provider can read prompts and activations the LeWorldModel processes. This makes provider admission a constitutional concern. The framework's discipline:

**1. Open self-registration.** Any DAC can register as a sovereign-compute provider via an open registry maintained by the SCC (the SCC itself is the default provider; other providers federate with it as peers, not competitors). Registration publishes:
- Hardware specs (GPU class, RAM, storage, network)
- Jurisdiction (legal home of the operator)
- K4 Grace Exit terms (data-deletion guarantee with cryptographic deletion proof on customer migration)
- Encrypted-summary protocol compliance attestation (per Q3 reconciliation)
- Self-declared η-class (any extraction is disclosed; η > 0 providers are admitted but rendered visible to users)

**2. Probationary tier T0.** New providers serve only DACs that explicitly opt into T0. Restrictions:
- Encrypted summaries only (no raw data; per Q3 reconciliation)
- Strict bandwidth caps
- Random canary-injected plaintext leak tests (the SCC injects fake plaintexts and monitors public surfaces for any leak)
- Limited compute share per DAC

**3. Reputation tiers T1, T2, T3.** Promotion based on durable receipts:
- **T1 (90 days, no incidents):** receipt-confirmed uptime ≥ 99.5%, zero-leak audits clean, p99 latency within target, K4 Grace Exit honored on every customer migration with cryptographic deletion proof
- **T2 (180 days at T1):** all of T1 + cross-customer benchmark (model performance does not degrade across multi-tenant load)
- **T3 (365 days at T2):** all of T2 + open-audit attestation (third-party audit of operator's data-handling practices, paid by SCC, results published over RELAY)

**4. Slashing for breach.** Any leak detected by canary, any K4 violation, or any η > 0 extraction beyond the self-declared η-class triggers immediate provider-wide slashing:
- All current customers receive K4 Grace Exit notification + automatic SCC re-routing within 24 hours
- Provider's reputation tier resets to T0
- Provider's API PAY revenue clawed back (FLOW receipt of clawback) to compensate affected DACs
- Public RELAY broadcast of the slashing event with evidence
- AXIOM market opens on whether the provider should be re-admitted (community decides)

**5. K2 acceptance gate per DAC, per provider.** Even an open-tier provider requires the DAC's K2 holder (private DAC) or PRISM governance act (public DAC) to *explicitly* accept the provider before the LeWorldModel routes any work to them. K2 acceptance is **per-provider** (not per-call) — a single tap authorizes a provider for the DAC's substrate. Switching providers later is a K4 Grace Exit + new K2 acceptance (single tap each).

**6. SCC default; user override always available.** A new DAC at instantiation defaults to SCC-curated assignment (per Q5 reconciliation). The user can opt out at any time and bring their own provider — including running their own GPU under their own keys (full self-hosting, no external admission process needed).

This closes the η > 0 risk Q10 named: providers admitted openly, but with reputation-graded probation; canary slashing for breach; K2 acceptance gate per-provider; user override always available. The compute-substrate-layer η = 0 commitment is preserved through the admission discipline, not through gatekeeping admission itself.
