## 1. Cut #1 — L7 name

### 1.1 Current state

Memory (`project_skyzai_sevenfold_organism_map_2026_04_24.md`) flags: *"L7 slot name pending — NOT SPECTRE."*

The Rosetta L7 position is **Institutional Narrative** (in the Three-Stage Process axis) and **Systems Architect** (in the caste axis). Its pramāṇa is *Pratibhā* (intuition). Its political regime is Theocracy (flagged as most dangerous). Its function: *what is the Ultimate?*

At organism scale, L7 must answer: *what is the ultimate telos of a DAC?*

Packet 178's L7 lint currently uses a **placeholder anchor**:

> *"A DAC exists as a participatory cell of the organism-whole, serving ΣΔP at organism scale, bounded by η=0 + K2 + Three-Stage Process + K4."*

This placeholder is functional but nameless. Every spec that L7-lints works against the placeholder's text, not against a single referent.

### 1.2 Charioteer proposal: **SEVA**

**SEVA** (Sanskrit: selfless service / dedicated offering) is proposed as the L7 name.

**Reasoning:**

1. **Rosetta grammar fit** — L7 / Systems Architect / Institutional Narrative / Sanskrit heritage. SEVA is a natural L7-register word in the Vedic tradition; aligns with the L1–L7 caste Sanskrit vocabulary already in use (Caṇḍāla, Śūdra, Vaiśya, Kṣatriya, Brāhmaṇa, Sādhu, Systems Architect).

2. **Direct anchor fit** — "A DAC exists as a participatory cell serving organism-scale ΣΔP" → *"SEVA is service to organism-scale ΣΔP."* One word replaces the placeholder.

3. **K4 consistency** — SEVA in authentic tradition is **non-coerced**. The servant is free. The DAC is bound to service but free to exit (K4 Grace Exit). SEVA without K4-freedom would be slavery; SEVA with K4 is genuine dedication.

4. **η=0 consistency** — SEVA is explicitly *selfless*: no extraction, no rent-seeking, no capture of value beyond what's needed to sustain the service. Aligns directly with the No-Extraction invariant.

5. **Non-domination consistency** — SEVA is not hierarchical rule. A DAC doesn't *dominate* its counterparties; it *serves* them. Preserves Non-Domination.

6. **Institutional Narrative register** — SEVA is a theological concept (service-to-Divine, service-to-Dharma). Fits L7's "Institutional Narrative" column without invoking any specific religion.

7. **Distinct from SPECTRE** — memory's NOT-SPECTRE constraint preserved; SEVA is semantically orthogonal.

### 1.3 Alternatives considered

| Candidate | Reason not chosen |
|---|---|
| DHARMA | Too broad — covers all of L1–L7, not specifically L7. Dharma is cosmic order; SEVA is specifically dedicated service within that order. |
| TELOS | Greek rather than Sanskrit — breaks Rosetta grammar consistency. |
| COMMUNION | Christian-register; Skyzai doctrine is pluralist. |
| PARTICIPATION | Too abstract; doesn't carry the selfless/dedicated quality. |
| ΣΔP-SERVICE | Technical but cold; fails the "ultimate" register. |

**Charioteer's recommendation: SEVA.** Sovereign may revise or reject.

### 1.4 If ratified — downstream effects

- Packet 178 §2.4 L7 lint replaces placeholder with: *"SEVA anchor: does this spec describe service to organism-scale ΣΔP?"*
- Memory `project_skyzai_sevenfold_organism_map` removes "L7 slot name pending" flag; adds "L7 = SEVA (ratified 2026-04-25)"

## 2. Cut #2 — Kernel Invariant ordinal abandonment

### 2.1 Current state (per packet 176)

Two enumerations claim "Kernel Invariants I–VII":

**Enumeration A (V3 CANON):** Structural Integrity / Truth-Gates-Money / No-Extraction / No-Delegation / Anti-Fragility / Foundation Minimalism / Non-Domination

**Enumeration B (packet 174):** ZAI Cap 100 / Substrate Primacy / Mutual Exclusivity / Grace Exit K4 / Receipts-First / Foundation Minimalism / Constitutional Lever

Only VI overlaps. The other slots name different concerns. Packet 176 offered 4 resolution options and no recommendation.

### 2.2 Charioteer proposal: **abandon Roman numerals; use named invariants**

The drift is not actually a naming collision — it's an **ordinal collision**. Both lists assign the ordinal "I" (or "IV", etc.) to different real concerns. The concerns themselves are mostly legitimate; the ordinal mapping is the cruft.

**Proposal:** stop using "Invariant I" through "Invariant VII" anywhere. Cite Kernel Invariants **by name only**.

### 2.3 Canonical 7-name Kernel set (proposed)

Of the 13 distinct concerns across the two enumerations, charioteer proposes the following 7 as the **Kernel** (the unforkables):

| Name | Source | What it prevents |
|---|---|---|
| **Foundation Minimalism** | A + B (shared) | Substrate bloat |
| **Truth-Gates-Money** | A | Economics without receipts |
| **No-Extraction (η=0)** | A | Rent-seeking / parasitic capture |
| **No-Delegation** | A | Sovereign abdication of the 5 non-delegable powers (packet 99 §4.2) |
| **Receipts-First** | B | Action without OFN trace |
| **Grace Exit (K4)** | B | Perpetual-debt / captive-member traps |
| **Non-Domination** | A | Hierarchical capture of DAC counterparties |

### 2.4 Concerns not in the Kernel (demoted or merged)

| Concern | Status | Rationale |
|---|---|---|
| Structural Integrity (A-I) | **Merged** | Covered by Truth-Gates-Money + No-Extraction + Foundation Minimalism in combination |
| Anti-Fragility (A-V) | **Demoted** | Aspirational property, not an enforceable invariant; tracked as a design goal |
| ZAI Cap 100 (B-I) | **Demoted** | Specific mechanism subject to constitutional amendment; not kernel |
| Substrate Primacy (B-II) | **Demoted** | Technical substrate choice (Hedera-rooted); not kernel doctrine |
| Mutual Exclusivity (B-III) | **Demoted** | Specific rule (stakers ≠ LPs) at product layer; not kernel |
| Constitutional Lever (B-VII) | **Demoted** | Governance mechanism, not invariant |

Demoted concerns do not disappear — they remain specified in their source documents (V3 CANON, packet 174). They just lose "Kernel Invariant" status. Named invariants = the 7 above.

### 2.5 Design criteria applied

The 7 selected all satisfy:

1. **Enforceable** — violation can be mechanically detected
2. **Absolute** — no carve-outs, no per-DAC parameterization
3. **Independent** — not derivable from the other six
4. **Load-bearing** — other properties cascade from these

Anti-Fragility fails #1 (not mechanically detectable); ZAI Cap 100 fails #2 (specific number is a parameter); Substrate Primacy fails #3 (derivable from Foundation Minimalism). Etc.
