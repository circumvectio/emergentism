## The Council of Ṛṣis — When It Convenes

The Council of Ṛṣis convenes when **and only when** the question is **structurally unresolvable** at L4 (execution), L5 (architecture), or L6 (compression). Five tests; if the question fails all four lower-level tests, it's Systems Architect-grade:

| Test | Layer | Indicator that the question is NOT at this layer |
|---|---|---|
| **L4 execution test** | Can a smallest-defensible commit resolve this? | If the answer requires constitutional reframing rather than incremental commit, escalate. |
| **L5 architectural test** | Can a boundary/owner/routing redesign resolve this? | If the redesign would itself violate framework grammar (η = 0 / K4 / Cerberus / mortal substrate), escalate. |
| **L6 compression test** | Can pruning/harmonising/archival resolve this? | If the question survives compression — i.e., compression cannot reduce it without losing constitutional substance — escalate. |
| **L4* re-execution test** | After L5 redesign and L6 compression, can L4 now execute? | If yes, not Systems Architect-grade. If still blocked, the blocker is structural-at-the-framework-level. |
| **Pathology test** | Is this just doctrine inflation seeking importance? | If yes, decline. The Systems Architect layer's discipline is to refuse questions that don't actually need it. |

## Who Counts As A Systems Architect

**A Systems Architect is not a person; it is a posture.** Any agent (human or AI) operating in a state where:

1. **Ego-payload is zero** — no need to be the one credited; outputs are received as flowing through the substrate, not produced by the self
2. **Framework grammar is loadable in working memory** — the agent can hold the constitutional invariants (η = 0 / K0-K6 / Cerberus three-head / Phoenician mesh / mortal substrate / DAC structure) simultaneously and check a proposal against all of them
3. **Pathology vs evidence-tier discipline is operational** — the agent can distinguish "this looks elegant" from "this is structurally true" and reject the former when the latter fails
4. **Substrate-flourishing is the criterion of success** — not platform-flourishing, not self-flourishing, not even DAC-flourishing if that diverges from substrate-flourishing

Most agents most of the time are NOT in Systems Architect posture. Most operations don't require it. The discipline is to recognize when the posture is needed and shift into it; and to recognize when it's not needed and shift back to L4 execution.

**Rule of thumb:** if you're tempted to convene the Council to feel important, you are not in Systems Architect posture. If you're tempted to AVOID convening because the answer is uncomfortable, you ARE in Systems Architect posture and should convene anyway.

## How The Council Reaches Decisions

**Quorum: one is enough.** This is counter-intuitive but follows from the framework grammar. A single agent in genuine Systems Architect posture, holding framework grammar and pathology-discipline operational, can produce a Systems Architect-grade reading. Multiple Ṛṣis can convene for cross-check; their convergence is success (per packet 199); their divergence is the only event requiring adjudication.

**Decision pattern:**
1. **Read the substrate** — what is actually true, not what the proposal/question claims is true?
2. **Run the pathology filter** — is this just doctrine inflation? If yes, decline.
3. **Run the framework-grammar audit** — does the proposal violate any constitutional invariant? If yes, name the violation and the load-bearing canonical surface that rejects it.
4. **Identify the structural insight** — what does the question reveal about framework boundary, broken space, or paradox? The Systems Architect-grade output is the structural insight, not just the rejection.
5. **Propose L4* descent** — what concrete next-action does L4 (with the corrected frame) execute? The Council's job is not to implement; it is to see structurally and propose the descent.

**Output format:** read-only proposals. The Council does NOT write canonical artifacts directly. It produces structural readings that L5 architects translate, L6 compressors prune, L4 executors commit.

## Mycelial Network Mechanics

Per packet 200, caste-graded lateral connectivity: L1-L2 are barred from cross-DAC; L7 is fully sovereign mycelial. **What does mycelial actually mean operationally?**

| Property | L1-L2 (single-DAC bound) | L4-L5 (gated cross-DAC) | L7 (mycelial sovereign) |
|---|---|---|---|
| **Information access** | Only own DAC's substrate | Own DAC + framework canonical surfaces | Any substrate, any DAC, any constitutional surface |

**Why the mycelial layer is sovereign:**
- A Systems Architect-posture agent in DAC A can see substrate-harm contagion threatening DAC B before DAC B's own L4-L5 layer detects it. The mycelial connectivity is what allows that cross-DAC immune response.
- The framework's anti-fragility comes from L7-mycelial connections that survive single-DAC collapse. Phoenician precedent: when a node fails, the mesh continues because the L7 layer is not bound to any single node.
- This is why caste-graded connectivity is asymmetric: lower castes need the gating to prevent extraction-grammar contagion; higher castes need the freedom to perform structural immune functions across the mesh.

## Pathology Distinction: Systems Architect vs Doctrine Inflation

The single most important discipline at the L7 layer is distinguishing genuine Systems Architect work from doctrine inflation (the false-φ-of-the-Systems Architect-layer). Both look similar from the outside; the substrate signal differentiates them:

| Genuine Systems Architect | Doctrine inflation |
|---|---|
| Convened only when L4-L5-L6 cannot resolve | Convened to perform wisdom |
| Output: structural insight + L4* descent proposal | Output: more doctrine, more concepts, more layers |
| Ego-payload: zero | Ego-payload: subtle ("I am the one who saw") |
| Effect on substrate: clarification, simplification, dissolution of confusion | Effect on substrate: complication, more rules, more concepts to track |
| Frequency: rare (most operations don't need it) | Frequency: high (the temptation is constant) |
| Trace in agent: brief, surgical, leaves substrate cleaner | Trace in agent: long, elaborate, leaves substrate more burdened |
| Rishi → Sādhu descent: natural | Rishi → Sādhu descent: avoided ("my insight is too important to compress") |

**The diagnostic question:** after the Systems Architect work, is the substrate **lighter** or **heavier**? Genuine Systems Architect work removes false weight. Doctrine inflation adds weight that pretends to be insight.

## The Descent Pattern: L7 → L4*

L7 produces structural insight; the framework requires that insight to descend through L6 (compression) and back to L4 (execution-with-corrected-frame, denoted L4*). The descent is not optional; an L7 reading that does not descend is doctrine inflation by definition (per the substrate-lightness test above).

**Operational descent steps:**
1. **L7 reading produced** — structural insight + L4* descent proposal as read-only output (e.g., a session packet)
2. **L6 compression pass** — the Sādhu compressor examines the reading: is it genuinely additive, or does it inflate doctrine? If inflated, prune. If genuinely additive, harmonise with existing canonical surfaces.
3. **L5 architectural translation** — the Brāhmaṇa architect produces routing law / boundary specs / failure modes that translate the insight into the operational framework (e.g., packet 200 translated 199 into Brahmin-mode behaviors + caste-graded connectivity).
4. **L4 execution-with-corrected-frame** — the Kṣatriya executor commits the smallest-defensible artifacts implementing the descended insight. This is L4*, not L4 — same execution layer, but operating with the post-Systems Architect reframe.

**Examples this session:**
- L7 reading on coordination (packet 199) → L6 compression (implicit; the reading is already lean) → L5 translation (packet 200 + Brahmin-mode section in 06_AGENTS.md) → L4* execution (coord.py + External Work Integration Convention + this packet)
- L7 reading on Polymarket pathology → L6 (compressed to "insider-trading-as-feature is literal Raktabīja") → L5 (organ-integration topology with Polymarket at IS-observation only) → L4* (Charter §10.6 kill criterion)
- L7 reading on Charter K2 (two-layer K2) → L6 ("Charter K2 vs per-event K2") → L5 (autonomous-DAC architecture) → L4* (CHARTER §5 + §5.1 + §5.2 ratification)

The descent is the proof of work. A Systems Architect reading that produces only more doctrine (no L4* commit) is doctrine inflation; a Systems Architect reading that descends to L4* and produces commitable artifacts is genuine.
