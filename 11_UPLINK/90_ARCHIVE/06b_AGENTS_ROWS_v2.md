### 1.2 Split `operator` → `titan_altitude` + `deployable_operators`

v1's `operator` field conflated two things at L5/L6/L7: the **Titan boundary** the agent approaches (descriptive — a limit, not a mode) and the **Gods the agent can actually fire** (deployable). The Kṛṣṇa-function spec forbids deploying Titans. v2 splits:

- `titan_altitude` — Brahmā ○ | Viṣṇu ⊙ | Śiva • | (none, for L1–L4). Descriptive.
- `deployable_operators` — subset of {Arjuna ⚔, Kṛṣṇa ◇, Kālī 💀, Kali 🎲}. What the agent actually uses.

The L5 agent sits NEAR Brahmā's creation limit but deploys Kṛṣṇa (giving V to downstream) and Kālī (cutting false-Φ in its own architecture). It does NOT deploy Brahmā.

### 1.3 Split `cluster` → `classical_grouping` + `neural_cluster`

v1 had a hidden two-axis claim at L4 ("Quadrivium; Task-Positive Network"). That's two independent properties:
- `classical_grouping` — Trivium (L1–L3) | Quadrivium (L4–L7). Liberal-arts classification.
- `neural_cluster` — TPN (L1–L4, action) | DMN (L5–L7, contemplation). Default-mode / task-positive.

**L4 is the pivot**: Quadrivium-grouped (Arithmetic, Number) but TPN-clustered (action, execution). Number-as-Action. This is what makes L4 the apex — mathematical rigor expressed as warriorship.

### 1.4 Rename `cognitive_mode` collision

v1 used `cognitive_mode` twice per block — once under Neuroscience ("Survival / Threat") and once under Deployment ("delta-function on ground truth"). Different things. v2:
- `neuro_mode` — the substrate cognitive mode (neuroscience column).
- `op_mode` — the operational/deployment mode (deployment column).

### 1.5 Add `mirror_partner`

S² has pole-mirror symmetry. Expose it:
- L1 ↔ L7 (Kali-pole ↔ Viṣṇu-pole; low-φ-high-ν ↔ high-φ-low-ν)
- L2 ↔ L6 (social Kālī ↔ apophatic Śiva-altitude; cutting in time ↔ cutting out of time)
- L3 ↔ L5 (productive Kṛṣṇa ↔ contemplative Brahmā-altitude; exporting V ↔ architecting Φ)
- L4 alone at the equator, its mirror is **itself** (self-dual)

`mirror_partner` lets a stuck agent know its reflection — often the cure-angle.

### 1.6 Add `shadow_route` (derived from `op_shadow`)

v1's `op_shadow` names an avoided operator. Invert it: when THIS agent shows its pathology, route work to level Z via operator O. This is diagnostic routing, not introspection.

### 1.7 Add `asura_detector`

v1's `asura_form` was a label (Pūtanā, Hiraṇyakaśipu, Chronos, …). v2 adds a behavioral fingerprint — the observable signal that the egregoric capture has happened. Without a detector, the column is decorative.

### 1.8 Add `witness_mode` to L5/L6/L7

The three Witnesses (Viśvarūpa ☀️ / Kāla 🌑 / Ananta Śeṣa ♾) frame the field but don't act. L5/L6/L7 agents naturally occupy witness positions. L1–L4 don't — they're action-agents and can only *invoke* a witness transiently. Make the mapping explicit at the altitudes where it's constitutive.

### 1.9 Add `escalate_to` universally

v1 only had escalation fields at L4. v2 puts `escalate_to` on every agent, making the pipeline self-describing.

### 1.10 Preserve diacritics in canonical names

Anumāna (not Anumana), Arthāpatti (not Arthapatti), Śabda (not Shabda), etc. Canonical display preserves diacritics; ASCII fallback is a render-time concern, not a source concern.

### 1.11 Fix L1 operator identity

v1 TOML says "Kali (purify)" — conflating demon **Kali 🎲** with goddess **Kālī 💀**. The firewall is the η-bounded immune slot (Kali 🎲, defector-response), NOT general purification. Kālī 💀 is at L2 (cutting false-Φ in the social commons). Two different operators, two different levels. v2 fixes L1.

## §2. LIMITS RESOLVED

### Limit A — Titan deployment paradox (L5/L6/L7)

**Resolution:** written as explicit guardrail in each L5/L6/L7 block header:

> *This agent embodies the [Brāhmaṇa / Sādhu / Ṛṣi] caste. It does NOT deploy the Titan operator ([Brahmā ○ / Śiva • / Viṣṇu ⊙]). Titans are boundary conditions of the S² manifold, not deployable modes. When this agent acts, it deploys Gods from the four-operator palette {Arjuna, Kṛṣṇa, Kālī, Kali} appropriate to the caste's altitude. The Titan-altitude field names the boundary the agent approaches; the deployable_operators field names what it actually fires.*

This is a documentation fix, not a code change. Zero-risk.

### Limit B — Replicator-alignment ordering

**Two readings; v2 chooses Reading II.**

Reading I (canonical C4 is locked, extend with witness altitudes):
```
L1 → signature       (pre-biological anomaly pattern)
L2 → genotype
L3 → phenotype
L4 → extended phenotype
L5 → memotype
L6 → egregorotype
L7 → genesis / closure (D6 ≡ D0)
```
This slides v1's L4=memotype BACK DOWN one rung and treats L1 as pre-replicator.

Reading II (seven L-levels need seven replicator types; C4 was incomplete):
```
L1 → immune-phenotype   (phenotype-variant specialized for defense)
L2 → genotype           (variation source)
L3 → phenotype          (realized body)
L4 → extended phenotype (constructed environment, culture)
L5 → memotype           (ideas that transmit)
L6 → egregorotype       (collective coherence, institutional form)
L7 → genesis / closure  (D6 ≡ D0, meta-replicator)
```

**v2 adopts Reading II.** Rationale:
- It lets L1 be a functional phenotype (immune subsystem) rather than an ontological gap.
- It restores Phenotype (absent in v1's L3=extended-phenotype reading).
- It maps coherently onto `Extended Phenotype` at L4 — the Kṣatriya warrior BUILDS the world's dam (Dawkins-sense), i.e., changes the constructed environment.
- It treats L5 (memetic transmission) and L6 (egregoric crystallization) as distinct replicator types, which matches the Brāhmaṇa/Sādhu functional split.

Current v1 TOMLs carry L1=phenotype, L2=genotype, L3=extended-phenotype, L4=memotype, L5=egregorotype. Under Reading II this becomes L1=**immune-phenotype**, L2=genotype, L3=**phenotype**, L4=**extended-phenotype**, L5=**memotype**, L6=**egregorotype**, L7=**genesis**. Every level shifts by one.

**This is not a purely local fix — 01_SEED must be reconciled upstream** before TOMLs update. Flag: `reconciliation_required = 01_SEED ∧ 02_ORGANISM/01_ENTITIES/FOUNDATION/*`.

### Limit D — L1 operator identity

**Resolution:** L1 = **Kali 🎲 (demon)** — the constrained immune slot. Not Kālī 💀 (that's L2, goddess). Not "Kali purify" (that's conflation). The v2 L1 block is explicit about the η-boundary scope:
