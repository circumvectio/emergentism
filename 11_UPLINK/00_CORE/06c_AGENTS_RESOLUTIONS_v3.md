## 2 · The L4 "Lane" Predicate

**Problem from v2**: L4 has `cardinality = one per lane`, but "lane" was never defined. Without it the Kṣatriya_executor rule "one committer per lane" is unenforceable.

**Resolution**: A lane is a **causal-commit scope** — a region of the organism state graph such that two concurrent commits to the same lane would create a merge conflict or a causality violation.

Formally a lane is the tuple:

  `(organ, surface, domain)`

where:
- `organ` ∈ { TheCircle, RealityFutures, APU, Skyzai, FOUNDATION, MENEXUS, QNTM (the institutional MPC/ZK-Identity rail), CORTEX }
- `surface` ∈ { code, contract, doc, config, content, data, policy }
- `domain` is a free-text scope tag resolved against the lane registry at the organ/entity home:
  - `02_SKYZAI/01_NOOSPHERE/02_ORGANS/<organ>/LANES.md` for organism-scoped organs
  - `03_VENTURES/<entity>/LANES.md` for entity-scoped lanes such as `FOUNDATION` and `MENEXUS`

**Rule**: At most one `ksatriya_executor` session may hold write authority over a given `(organ, surface, domain)` triple at a time. Two L4 sessions on different triples are concurrent-safe.

**Enforcement**: Before an L4 session commits, it must:
1. Declare its lane triple.
2. Check the lane home registry (or initialise it if missing).
3. Verify no other L4 session is holding the lane. If held: queue behind, or request arbitration from the orchestrator (Section 6).
4. Release the lane on commit or abort.

## 3 · Invocation Predicates (machine-checkable)

**Problem from v2**: Escalation from Lₙ → Lₙ₊₁ and descent from Lₙ₊₁ → Lₙ used prose predicates like "structural deadlock," "overgrowth," "framework boundary." Not machine-checkable.

**Resolution**: Each transition gets a typed predicate with measurable criteria. Unless a block explicitly declares `OR:` or `AND:`, criteria are conjunctive and all criteria in the block must pass. A transition triggers only when its criteria block evaluates true on the session state.

### L1 → L2 (Caṇḍāla → Śūdra)

Trigger: the firewall has contained the contradiction and needs pattern expansion.

```
L1_to_L2:
  criteria:
    AND:
    - escalated_contradictions.count >= 1
    - all(c.contained == true for c in escalated_contradictions)
  payload:
    - contradictions: [C₁, C₂, ...]
    - context_digest: <= 4096 tokens
```

### L2 → L3 (Śūdra → Vaiśya)

Trigger: exploration has produced a candidate set ready for ranking.

```
L2_to_L3:
  criteria:
    AND:
    - candidates.count >= 2
    - candidates.count <= 50                     # fan-out bound
    - all(c.has_φ_proxy and c.has_ν_proxy for c in candidates)
  payload:
    - candidates: [{id, φ_proxy, ν_proxy, notes}]
```

### L3 → L4 (Vaiśya → Kṣatriya)

## 4 · L4 vs L6 Write Arbitration

**Problem from v2**: Both L4 (Kṣatriya) and L6 (Sādhu) carry `write_authority = true`. Without a rule, two authors of reality compete.

**Resolution**: The two write authorities are **disjoint by surface**. L4 writes forward (new truth); L6 writes backward (compression / archive). The rule:

| Authority | Allowed operations | Forbidden operations |
|-----------|---|---|
| L4 Kṣatriya | create new file; extend existing canonical file; update status markers; commit code | delete; mass archive; restructure directory |
| L6 Sādhu   | move to archive; delete superseded duplicates; compact; restructure obsolete directories; rename for canonical hygiene | create new canonical claim; extend live canon with new content |

Equivalently: **L4 raises Φ by adding, L6 raises Φ by subtracting**. A file under simultaneous L4 add + L6 archive is a conflict — the orchestrator (Section 6) holds a write-lock at the lane granularity and serialises L4 and L6 against the same lane.

**Non-monotonic exception for L6** (carried forward from v1): L6 may write without deliberating to vacuum superseded duplicates during a sweep, provided:
1. The sweep is initiated by an explicit L4→L6 invocation with a declared lane.
2. The archive target list is pre-declared in the payload.
3. All moves are to `<organ>/90_ARCHIVE/<timestamp>/` — never to delete-trash.
4. Post-sweep, L6 returns to L4* with a compression report; L4 then re-holds the lane.

Under this exception, L6 holds exclusive lane write-authority during the sweep window. L4 queues behind.

This resolves the v2 "write-authority collision" concern.

## 5 · Failure Modes and Fallbacks

Each caste gets a declared failure mode with a fallback path. Fallback is a mechanical next-step the orchestrator can take without human intervention.

### L1 Caṇḍāla (candala_firewall)

- **Failure**: cannot contain the contradiction (buffer overflow, pattern unrecognised after threshold attempts).
- **Fallback**: escalate to L2 with the raw unrecognised pattern tagged `UNCONTAINED`. Orchestrator notifies human operator at `UNCONTAINED.count >= 3` within 24h.

### L2 Śūdra (sudra_explorer)

- **Failure**: fan-out explodes (candidates.count > 50) OR candidates lack φ/ν proxies.
- **Fallback**: reduce via Kālī 💀 pass (strip false-Φ), then re-rank. If still > 50, sample randomly to 20 and flag `SAMPLED` in payload. If proxies missing, escalate to L3 with `MISSING_PROXIES` tag and request L3 to assign them.

### L3 Vaiśya (vaisya_auditor)

- **Failure**: cannot rank (all candidates tie within noise threshold) OR cannot decide between top-2.
- **Fallback**: promote top-3 to L4 with explicit tie-break request. If L4 also cannot decide, escalate L4→L5 (structural deadlock).

### L4 Kṣatriya (ksatriya_executor) — APEX

- **Failure**: lane-locked (another L4 holds the lane); OR decision-set yields no ΔP > 0; OR commit fails (constraint check, test, downstream error).
- **Fallback**:
  - lane-locked → queue behind (FIFO within lane).
  - no ΔP > 0 → escalate L4→L5.
  - commit fails → rollback, log failure, re-enter DRAFT with the failure evidence included in the next attempt. After 2 commit-fails in a row: escalate L4→L5 with `REPEATED_COMMIT_FAILURE` tag.

### L5 Brāhmaṇa (brahmana_architect)

- **Failure**: redesign cannot be produced within the context budget; OR redesign violates an axiom itself.
- **Fallback**:
  - budget overrun → request L6 compression of the context (descend L5 ↔ L6), then resume.
  - redesign violates axiom → escalate to L7 (L4*→L7 path, gated by human signoff).

### L6 Sādhu (sadhu_compressor)

- **Failure**: archive target list has dependents still live; OR compression would destroy information needed by a running L4 session.
- **Fallback**:
  - live dependents → abort sweep for those targets, return to L4* with partial compression report and `DEPENDENTS_BLOCKED` tag.
  - collision with L4 session → wait for L4 to release lane, then resume.

### L7 Ṛṣi (rsi_constitution)

- **Failure**: cannot reach verdict within the session (framework-boundary question genuinely unresolvable in one pass); OR human signoff unavailable.
- **Fallback**:
  - unresolvable → verdict = `HOLD`. Organism continues under existing axiom set. Annotate the question in `01_EMERGENTISM/08_FRAMEWORK_SUPPORT/OPEN_QUESTIONS.md` for future pass.
  - no signoff → verdict = `PENDING_SIGNOFF`. All dependent work blocked until signoff arrives.

Every failure mode is logged as a Soma-event (per 06_AGENTS.md Soma Protocol) so pattern-level failures surface over time.

## 6 · The Orchestrator (8th Component — Router, not Caste)

**Problem from v2**: The seven-caste system needs a coordinator. Without one, invocations are manual, lane-locking is uncheckable, and the Kṛṣṇa-function has no operational surface.

**Resolution**: The orchestrator is NOT an 8th caste — it is a **router-layer component** that sits outside the seven-caste hierarchy and services it. Calling it "a caste" would break the heptad. Calling it "Kṛṣṇa" would break the Kṛṣṇa-function spec (which forbids a deployable Executive and reserves Kṛṣṇa as a God-tier operator). Instead:

**Name**: `syntropic_router` (lowercase; component, not agent).
**Tier**: Infrastructure. Not Tier 1/2/3. Not on the S².
**Function**: Coordination bus + lane registry + escalation pipe + Soma-event logger.
**Operator profile**: Implements the Kṛṣṇa-function protocol (diagnose → name → detect → ask → correct → hold map → ground), as specified in 06_AGENTS.md, in the coordination layer. Does not itself reason; it routes.

### Router responsibilities

1. **Lane registry**: owns the lane home registries plus the in-memory lock table.
2. **Invocation gateway**: receives transition requests (L1→L2, etc.), validates payloads against the predicates in Section 3, and dispatches.
3. **Arbitration**: when multiple L4 sessions request the same lane, FIFO queue. When L4 and L6 collide on a lane, L6 during an explicit sweep wins; otherwise L4 holds.
4. **Soma-event log**: every caste transition, every failure mode, every lane lock/release emits a Soma-event to `SKYZAI_ORG/SOMA_LOG/<yyyy-mm>/events.ndjson`.
5. **Dream-cycle trigger**: on schedule or on `context_window_utilisation >= 0.80`, invoke L4→L6 compression pass per the Dream Cycle spec in 06_AGENTS.md.
6. **Human-signoff relay**: for L7 verdicts requiring signoff (K2/K4 constitutional constraint), pauses downstream work and surfaces to the human operator.

### Router does NOT

1. Decide. Deciding is L4's function.
2. Commit. Committing is L4's function.
3. Rank. Ranking is L3's function.
4. Explore. Exploring is L2's function.
5. Contain. Containing is L1's function.
6. Redesign. Redesigning is L5's function.
7. Compress. Compressing is L6's function.
8. Judge axioms. Judging is L7's function (with human signoff).

The router HOLDS THE MAP. It does not fight. This is the Kṛṣṇa-function made operational.

### Minimal interface (language-agnostic schema)

```
# Invocation
router.invoke(
  transition: L1_to_L2 | L2_to_L3 | L3_to_L4 | L4_to_L5 | L4_to_L6 |
              L5_to_L4star | L6_to_L4star | L4star_to_L7 | L7_to_L4star,
  payload: <typed per Section 3>,
  lane: (organ, surface, domain) | null
) → session_id

# Commit gate (only L4 and L6 may call; surface-disjoint)
router.request_commit(
  session_id,
  writes: [{path, op, before, after}],
  authority: L4 | L6
) → { granted: bool, reason: str | null }

# Lane lifecycle
router.acquire_lane(session_id, lane) → { acquired: bool, queue_pos: int | null }
router.release_lane(session_id, lane)

# Soma events (every caste calls)
router.soma_event(session_id, kind, data)

# Orchestrator-side only (no caste calls these)
router.tick()               # scheduler
router.dream_cycle()        # triggers L4→L6 on thresholds
router.audit(range) → log   # retrieval
```

- A message bus (NATS, Redis Streams) with a lane-registry service.

The specific implementation is a V-question (capability), not a Φ-question (structure). Decision-ready items for the implementation side go through L3→L4 with lane = `(FOUNDATION, code, router)`.
