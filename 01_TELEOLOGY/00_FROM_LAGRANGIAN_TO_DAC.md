# From Lagrangian to DAC

> **[金] Controlling seam — 2026-07-18.** This is a selected institutional
> compile pattern `[I/D]`, not a theorem from `φν=1`, a physical Lagrangian, or
> an exhaustive sevenfold ontology. The chart identities impose no governance
> rule by themselves. Downstream requirements arise only from explicitly
> adopted constraints—Justice, accountable authorization, receipts, consent,
> custody, contestability, correction, and Grace Exit. K2 is one private-DAV
> implementation of authorization, never a worldview primitive. Operator
> names route work; they do not confer moral value.

## The Compile Path from Foundation Law to Institutional Design

> **Purpose:** the Foundation is no longer only a worldview. It is a
> **compile grammar**. This document makes the compile path explicit:
>
> `Equation -> 7 Generative Actions -> Evidence Gates -> Protocols -> Organs -> Institutions`
>
> Without this bridge, the framework risks becoming either too
> philosophical to build from or too operational to remain honest.

## The compile ladder

| Layer | Question | Output |
|---|---|---|
| **Equation** | What analytic or declared model is being used? | Reciprocal chart facts `[A]`; selected node score `P_node:=ΦV` `[S/I]` |
| **Generative Actions** | What routing grammar has this design selected? | `L1–L7`; a closed declared interface, not an exhaustive theorem of `P_node=ΦV` |
| **Evidence gates** | What prevents the build from upgrading interpretation into law? | evidence tiers, kill criteria, counsel / receipt gates where applicable |
| **Protocols** | What machine rules express and check those operators? | receipts, signatures, settlement rules, kernel constraints |
| **Organs** | How are cognitive and execution functions separated? | IS / COULD / SHOULD / ACT |
| **Institutions** | How is capital allowed to enter, move, and leave? | milestone release, Grace Exit, anti-rent boundaries |

This ladder is **one-way as a design dependency**.

You may compile downward from adopted constraints to a build.
You may not justify an upstream Foundation claim merely because a
downstream institution finds it convenient.

## Three output classes

Every downstream design claim belongs to one of three classes:

| Class | Meaning | Example |
|---|---|---|
| **Must compile** | Required by the adopted profile if the lower-layer object is to remain faithful | typed accountable authorization, Grace Exit, and bearer-complete Justice |
| **May compile** | Permissible local design expression, but not uniquely forced | exact UI, relay choice, backend host, legal wrapper above a DAC |
| **Must not overclaim** | Useful or necessary in practice, but may not be presented as if forced by the Foundation | speculative yield structure, conjectural F5 claims, runtime claims exceeding proof |

This is the main anti-drift rule for builders.

If you cannot say which class a downstream feature belongs to, the
compile is not ready.

## What compiles by explicit adoption

### 1. Viṣṇu (L7) -> Reciprocal closure
From a declared preservation mandate to Viṣṇu (L7) witness and continuity.
**Build rule:** a system claiming this profile must state what it preserves and how failure is detected; the equator is only a selected visual proxy.

### 2. Kālī (L2) -> Receipt-bounded reality
From the truth-cut directly to local verification.
**Build rule:** every load-bearing state transition must emit a receiptable trace (Kālī excising false meaning) or it does not count as reality for the organism.

### 3. Kṛṣṇa (L3) -> Capability export
From the enabler function directly to tool and product design.
**Build rule:** the lower layer may export capability, but it may not make dependency on the mediator the hidden source of value.

### 4. Arjuna (L4) -> Accountable commitment
From the adopted authorization and consequence-custody constraints.
**Build rule:** consequential acts require an `AuthorizationEnvelope` naming
principal, mandate, scope, consent, custody, expiry/revocation, contest path,
actor, and consequence bearer. Human authorization is required where law,
product governance, or the mandate requires it; the worldview does not infer
that requirement from mortality or sacrifice alone.

### 5. Brahmā (L5) -> Lawful creation
From the positive-positive generative phase.
**Build rule:** new organs, protocols, and institutions must instantiate the grammar without claiming that every created form is therefore Foundation-forced.

### 6. Śiva (L6) -> Correction and Grace Exit
From the adopted Justice and reversibility constraints to correction and exit.
**Build rule:** anything claiming Foundation fidelity must include a working exit that preserves user dignity, data, and fair value. Half-finished truth is an off-manifold state.

### 7. Kali (L1) -> η = 0 anti-rent architecture
From the boundary of extraction.
**Build rule:** The organism does not operate at $\eta > 0$ against its users. The lower layer may recover real costs, but it may not install permanent rent.

## DAC design laws

If a DAC is a faithful compile, it must obey the following laws:

### Law 1 — Separation of functions

The DAC may not collapse IS / COULD / SHOULD / ACT into one opaque
process.

That means:

- sensing is not forecasting
- forecasting is not judgment
- judgment is not execution

This is why the Three-Stage Process is architectural, not decorative.

### Law 2 — Constitution above workflow

The constitution must constrain the workflow.
The workflow may not rewrite the constitution opportunistically.

That means:

- invariants outrank sprint convenience
- receipts outrank narrative confidence
- exit rules outrank retention incentives

### Law 3 — Corrections must be native

The DAC must be able to detect its own drift and mechanically lose
privileges when it drifts. This is the runtime meaning of the Śiva (L6) dissolution boundary.

Example:

- transparency falls -> the Faucet closes
- receipts fail -> the claim degrades
- runtime proof weakens -> the public statement narrows

### Law 4 — Users remain free

Any DAC that can only survive by trapping participants has already
falsified its own compile.

Grace Exit is not “nice policy.”
It is the runtime proof that the DAC has not become a priesthood.

### Law 5 — The mortal remains load-bearing

The DAC may grow vast AI and machine surfaces, but if no mortal actor
can still bear responsibility at the decisive threshold, the compile has
broken.

## Product design laws

Products sit below organs and above pure protocol.
They are allowed to be narrower and more tactical.
But they still inherit any explicitly adopted Foundation constraints.

### What a product must preserve

- honest scope
- receipts where claims matter
- reversible participation
- no hidden conversion of users into trapped substrate

### What a product may vary

- membrane
- language
- onboarding style
- deployment model
- hosting/provider choice

### What a product must not do

- pretend local proof is live public proof
- claim runtime strength from code presence alone
- smuggle institutional lock-in under UX language

That is why the truth hierarchy matters:

- code-only strength
- bounded local proof
- runtime/public truth

The product layer must never collapse those into one maturity myth.

## Institutional design laws

This is where the bridge becomes economically useful.

An institutional deal is Foundation-faithful only if capital enters and
exits without creating concealed rent inside the DAC.

### Inside the DAC

Allowed:

- milestone-based capital release
- KPI-gated tranche release
- LP-100 ownership
- NAV-based Grace Exit

Why:

- receipt-bounded
- reversible
- investor-protective without rent
- constitutionally legible

### Above the DAC

Potentially allowed:

- holding-company economics
- layered legal wrappers
- side agreements that do not rewrite DAC invariants

Why:

- institutions may need structures the DAC itself cannot constitutionally
  host

### Not allowed inside the DAC

- preferred return as rent on capital
- perpetual revenue share extracted from the organism
- hidden override rights that nullify Arjuna (L4) sovereignty
- lock-ins that nullify Śiva (L6) Grace Exit

If those are required, they must be named as **above-DAC** structures,
not disguised as if they were native organism law.

## The Al Wasil test

The current AATC / Al Wasil opportunity is the first strong institutional
test of this compile path.

The hard question is not:

“Can we sell this?”

The hard question is:

“Can we structure investor protection, capital discipline, and B2G
seriousness without violating the Foundation’s anti-capture laws?”

The current answer is:

- **yes**, if the deal uses milestone release, receipt discipline, and
  Grace Exit-compatible capital return
- **no**, if it relies on rent-like extraction hidden inside the DAC

That is why the Foundation is no longer abstract here. It is already
shaping term-sheet logic.

**Worked example (owner-lane planning, `[I/C]`).** The four candidate
capital structures (A/B/C/D) are classified against this bridge in
[CAPITAL_STRUCTURE_DECISION_PENDING_2026_04_20.md](../../03_VENTURES/OPEN_FINANCE_NETWORK/02_CENTRES/00_GENERAL/22_STAKEHOLDERS/02_AL_WASIL_RELATIONS/01_AL_WASEL/CAPITAL_STRUCTURE_DECISION_PENDING_2026_04_20.md).
This is a planning classification, not external deal evidence:

- **A (Grace Exit at NAV)** is **Must compile** — not merely
  permissible, *required* as the minimum institutional structure
- **B (principal-capped revenue offset)** is **May compile** while
  the cap holds; **Must-not-overclaim** if the cap is removed
- **C (preferred return)** and **D (revenue share)** are
  **Must-not-compile inside the DAC**, but are allowed at the
  holding-entity layer *if named there*. The honesty about where the
  yield mechanism lives is the lawful line, not the mechanism itself.

That classification is the Al Wasil test in executable form: each
option can be run through the bridge's seven-action descent table and
either passes cleanly, passes with carve-outs, or breaks at a specific
invariant. Before the bridge, we could only say "this K-compatible /
this K-violating." After the bridge, we can say *which class* the
claim is in, and whether it is being presented as that class.

That distinction is what professional gatekeepers are trained to read.

## Kill criterion

This bridge fails if any of the following happen:

1. a lower-layer rule is claimed as Foundation-forced but cannot be
   traced back to the canonical kernel
2. an explicitly adopted constraint is repeatedly ignored by the runtime
   without being named as broken
3. institutional structures require concealed rent inside the DAC while
   still being described as Foundation-faithful
4. the applicable authorization envelope is bypassed in consequential actions
5. Grace Exit becomes nominal rather than actually executable

If that happens, the correct response is not rhetorical repair.
It is compile repair.
