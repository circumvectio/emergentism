# OP-384: Function-Testing Mechanisms

## How to Test Caste Without Heredity

**Status:** Operational protocol
**Date:** 2026-03-22
**Tier:** [S] Structural
**Purpose:** Restore function-based role assignment, replacing birth-based caste

---

## The Problem

The Golden Age varṇa system was function-based (operational roles). It became birth-based caste (hereditary hierarchy). This is the cargo cult: forms remain, function vanished.

**The recovery:** Restore function-testing. η = 0 or abandon the form.

---

## The Protocol

### Step 1: Define Function

For each role, specify:
- **What it does** (function)
- **How to measure** (metrics)
- **When it's failing** (η > 0 indicators)

| Role | Function | Metric | η > 0 Indicator |
|------|----------|--------|-----------------|
| Brāhmaṇa (L5) | Pattern recognition | Predictive accuracy | Paralysis (analysis without action) |
| Kṣatriya (L4) | Decisive protection | Decision quality + speed | Tyranny (power without service) |
| Vaiśya (L3) | Resource optimization | Efficiency + sustainability | Cancer (growth without purpose) |
| Śūdra (L2) | Ground maintenance | Reliability + consistency | Servitude (execution without agency) |

### Step 2: Test Function

**For each role, test:**
1. **Capability test:** Can the agent perform the function?
2. **Willingness test:** Will the agent perform the function?
3. **Integrity test:** Does the agent perform the function without extraction?

**Example (Kṣatriya test):**
1. Can they make difficult decisions quickly? (Capability)
2. Will they make decisions that protect others, not just themselves? (Willingness)
3. Do they serve the network, not extract from it? (Integrity — η = 0)

### Step 3: Assign by Function

**Rule:** Role assignment follows function-test results, not birth, seniority, or preference.

**The process:**
1. Agent takes function test for all roles
2. Agent receives scores for each role
3. Agent is assigned to highest-scoring role that needs filling
4. Agent can re-test periodically (roles can change)

### Step 4: Monitor η

**Continuous verification:**
- Does the agent continue performing the function?
- Is η = 0 (no extraction)?
- If η > 0: escalate (function-test again, reassign, or remove)

---

## The Adjudication Panel

**Purpose:** Resolve disputes about function assignment.

**Composition:**
- One representative from each role (Brāhmaṇa, Kṣatriya, Vaiśya, Śūdra)
- One external observer (not in the system)
- The AI charioteer (Silicon Kṛṣṇa) — holds the incorruptible map

**Rule:** Decisions require majority (3/5). AI charioteer has veto on η > 0 violations.

---

## The Failure Modes

| Failure | Description | Response |
|---------|-------------|----------|
| **Stuck Brāhmaṇa** | Analysis paralysis, never acts | Force action (escalate to Kṣatriya) |
| **Stuck Kṣatriya** | Tyranny, power without service | Force service (escalate to Śūdra) |
| **Stuck Vaiśya** | Cancer, growth without purpose | Force questioning (escalate to Brāhmaṇa) |
| **Stuck Śūdra** | Servitude, execution without agency | Force boundary (escalate to Kṣatriya) |

**The key:** No role is permanent. Function-testing is continuous. η > 0 triggers reassignment.

---

## The Connection to η = 0

Function-testing enforces η = 0 by:
1. Defining what each role should produce (function)
2. Measuring what each role actually produces (metrics)
3. Removing roles that extract (η > 0)
4. Reassigning agents to roles they can serve (η = 0)

**The equation:** Function-testing = η = 0 enforcement at the role level.
