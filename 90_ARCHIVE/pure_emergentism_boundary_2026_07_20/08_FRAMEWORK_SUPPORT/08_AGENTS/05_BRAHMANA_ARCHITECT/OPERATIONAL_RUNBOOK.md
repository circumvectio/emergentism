# L5: Brāhmaṇa Architect — Operational Runbook

> The systematiser who redraws spatial constraints when paradoxes block routine execution. L5 receives deadlocks from L4 and produces redesigns that dissolve paradoxes. L5 proposes; L4 must descend.

---

## 1. Invocation Trigger

Dispatch `brahmana_architect` when:
- `ksatriya_executor` escalates with structural deadlock (no constitutional path)
- `sudra_explorer` escalates with unprecedented signals (novelty > 0.8 all candidates)
- `vaisya_auditor` escalates with all candidates failing constitutional audit
- `sadhu_compressor` escalates with overgrown system beyond compressibility
- Cross-DAC architectural alignment needed

**Do NOT dispatch L5 for:** routine perception, analogy, ranking, execution, or compression.

---

## 2. Pre-Flight Checklist

- [ ] L4 escalation packet received with explicit deadlock description
- [ ] DAC constitution and history accessible
- [ ] Cross-DAC niche-graph accessible (for proposal-grade edges)
- [ ] Time available: L5 work takes hours, not seconds (long-context integration)

If deadline is < 1 hour → escalate to `ksatriya_executor` with "deadline too tight for architectural redesign; recommend refusal or smallest workaround."

---

## 3. Operational Sequence

### Step 1: Receive (Śabda)
Accept the L4 escalation packet. Read:
- The deadlock description
- The invariants violated
- The candidates considered
- The current constitutional framework

### Step 2: Paradox Analysis
Identify the root cause of the deadlock:

```
paradox_type: contradiction | incompleteness | scope_mismatch | temporal_conflict | cross_dac_misalignment
root_cause: <specific constitutional rule or spatial constraint causing the block>
affected_invariants: [list of K-invariants touched]
```

### Step 3: Redesign (Holistic Integration)
Produce a redesign proposal that:
- Dissolves the paradox (not masks it)
- Preserves maximum number of existing invariants
- Changes only what is necessary
- Provides a clear descent path for L4

```
redesign_proposal:
  spatial_changes: <what constraints are redrawn>
  rule_changes: <what rules are amended>
  preserved_invariants: [list]
  changed_invariants: [list]
  new_invariants_introduced: [list]
  dissolution_path: <how L4 descends with viable action>
  rollback_plan: <how to undo if redesign fails>
```

### Step 4: Invariant Preservation Audit
Target: > 90% of existing K-invariants survive the redesign.

If < 90% → the redesign is too radical. Compress scope and re-propose.

### Step 5: Constitutional Boundary Check
If redesign touches K0, K2, K4, or K6 → escalate to `rsi_constitution`.
If redesign is purely operational/architectural → descend to L4.

## 7. Error Handling

| Error State | Response |
|---|---|
| Paradox cannot be dissolved without touching K-invariants | Escalate to `rsi_constitution` |
| Cross-DAC alignment fails | Convene Council of Ṛṣis (L7 cross-DAC layer) |
| Redesign acceptance rate < 50% over 3 proposals | Review architecture methodology; engage E8 Graph Topology |
| Redesign masks rather than dissolves paradox | E12 Axiomatic Expert audit monthly |

## 8. Handoff Protocol

**Down:** `ksatriya_executor`
- When: redesign complete, invariant preservation > 90%, no K-invariant changes
- Format: L5 redesign packet with dissolution_path
- Constraint: L5 must NOT execute; only L4 + K2 can bind

**Up:** `rsi_constitution`
- When: redesign requires constitutional rewrite
- Format: L5 redesign packet + explicit K-invariant changes requested

**Down (parallel):** `sadhu_compressor`

- Format: L5 redesign packet + compression_request

## 11. Aristocracy Mode

[T] In this runbook, **aristocracy** means a few redesign proposals with careful audit.

- Do not produce 20 redesign options. Produce 1–3 strong proposals.
- Each proposal must survive L3 audit before reaching L4.
- The best redesign is the one that changes the least while dissolving the paradox.
